Project brief — MedVault

Build a local-first personal health records app

A mobile application that gives patients full ownership over their fragmented medical history — prescriptions, reports, images — stored entirely on-device with AI-powered summaries.

Problem context

Doctors treating patients in emergencies or follow-up visits across different hospitals cannot access consolidated medical histories, previous prescriptions, drug allergy information, or ongoing medication regimens — because health records remain fragmented across individual hospitals with no interoperable national health database system. This app puts the patient in control by consolidating all records on their own device.

Feature specifications
P1 — Core Prescription logging
Camera capture flow
User opens camera within the app, photographs a physical prescription. The image is saved to local storage at a deterministic path (e.g. medvault/prescriptions/{visitId}.jpg). OCR runs on-device (via ML Kit or Tesseract WASM) to extract text and pre-populate the medicine list. User can correct extracted fields inline before saving.
Manual entry form
A structured form with fields: doctor name, doctor phone / hospital number, date of visit, medicine name + dosage + frequency + duration (repeatable rows), diagnosis / notes free-text, next appointment date. All fields optional except date.
Doctor & hospital info
Doctor contact block is a reusable component: name, phone, hospital name, specialty. Previously used doctors are suggested via local autocomplete (stored in a separate contacts.json) so returning users aren't re-typing every visit.
P1 — Core AI prescription summary
On-device or local-API summary
Each prescription gets an AI-generated plain-language summary. Since data must not leave the device, use a locally-running LLM (e.g. Phi-3-mini via ONNX Runtime or llama.cpp) or a user-configurable private API endpoint (self-hosted Ollama). Summary covers: what the medicines are for, key side effects to watch, interactions flagged if multiple medicines present, and what to tell the next doctor. Summary is cached in JSON and regenerated only when the prescription changes.
Summary display UI
Shown as a collapsible card beneath the prescription. Three sections: "What you were prescribed", "Why (diagnosis summary)", "What to watch for". Uses plain language — no medical jargon. A small "regenerate" button re-runs inference if the user edits the prescription. Timestamp shown for when summary was last generated.
P1 — Core Timeline & sorting
Chronological linked list traversal
The visit linked list is sorted by date descending (most recent first) as the default view. Each card in the timeline shows: date, doctor name, hospital, top-line AI summary sentence, and a count of attached media. Tapping opens the full visit detail screen. Users can also sort ascending (oldest first) for longitudinal review.
Filter & search
Filter by: date range, doctor name, tag, media type present (has X-ray, has report, etc.). Full-text search across prescription text, doctor names, and AI summaries — implemented with a local inverted index (e.g. Fuse.js or MiniSearch) built from the JSON store at app launch.
P2 — Important Media attachments
Attachment types supported
X-rays, lab reports (PDF or image), before/after photos, general medical images. Each attachment stores: local URI, capture timestamp, type enum, and a free-text caption. Files stored at medvault/media/{visitId}/{mediaId}.ext. PDF viewer built-in (react-native-pdf or equivalent). Image viewer with pinch-to-zoom.
Media gallery per visit
A scrollable grid of thumbnails within each visit detail screen. Tapping opens a full-screen viewer with swipe navigation between attachments. Long-press shows caption + type. Users can add new attachments to any past visit at any time.
P3 — Nice to have Export & sharing
One-tap visit summary export
Generate a PDF of a single visit (prescription + AI summary + attached images) for sharing with a new doctor. Rendered entirely on-device. User explicitly chooses what to include before exporting. No auto-share; user controls the share sheet manually.
Storage architecture
JSON store layout
medvault/data/visits.json — master linked list of all visit nodes

medvault/data/contacts.json — doctor/hospital autocomplete cache

medvault/data/index.json — full-text search index, rebuilt on launch

All files written atomically (write to temp, rename) to prevent corruption.
Media file layout
medvault/prescriptions/{visitId}.jpg — prescription photos

medvault/media/{visitId}/{mediaId}.ext — all other attachments

URIs stored as relative paths in JSON; resolved to absolute on read. Prevents breakage on app reinstall if data is backed up manually.
Privacy & security requirements
Zero network calls for medical data
No analytics SDKs, no crash reporters, no cloud sync. The app operates fully offline. If an AI model requires a network call, it must be user-configured and explicitly opt-in with a clear warning that data will leave the device. All AI inference defaults to on-device.
App-level lock
Biometric (Face ID / fingerprint) or PIN lock on app open. Configurable auto-lock timeout. Data files should sit in the app's private sandbox (not accessible to other apps or the system media gallery).
Tech stack recommendation
Recommended
Framework: React Native (Expo) — cross-platform iOS + Android from one codebase

Storage: expo-file-system + custom JSON read/write layer

OCR: ML Kit Text Recognition (on-device, free)

AI summary: llama.cpp via react-native-llama.rn, or Phi-3 ONNX

Search: MiniSearch (lightweight, no server)

PDF: react-native-pdf for viewing, react-native-html-to-pdf for export
Constraints
No Firebase, Supabase, AWS Amplify, or any BaaS

No Mixpanel, Amplitude, Sentry, or analytics

No push notifications that require a server

Expo Go compatible for fast dev iteration; eject to bare workflow only if llama.cpp requires it

Target Android 10+ and iOS 15+
